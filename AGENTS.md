## Обязательные правила экранов

- У каждого экрана уникальный `screen_id`.
- У каждой кнопки, кликабельной строки, карточки и пункта меню есть `action_id`.
- У каждого `action_id` есть существующий экран или объявленное `system.*` действие.
- У каждого stack-экрана есть `back_fallback`.
- Для экрана описаны loading, empty, error, offline и permission denied, когда применимо.
- Публичный экран сущности не дублируется для Главной, Событий, Календаря, Чатов и Профиля.
- Управление сущностью находится на каноническом маршруте `/manage`.

## Обязательные правила UI

- Использовать размеры и компоненты только из `docs/DESIGN_TOKENS.yaml`.
- Не создавать локальную копию Button, IconButton, FilterChip, StatusBadge, Card, Header или BottomTabBar.
- Тёмная тема обязательна; hard-coded Hex/RGB внутри экранов запрещены.
- До утверждения палитры использовать только semantic color tokens.
- Минимальная зона нажатия на iOS и Android — 48×48.
- Если чипы не помещаются, использовать одну горизонтально прокручиваемую строку без wrapping.
- Основные продуктовые иконки брать из `lucide-react-native`; не смешивать большие icon libraries.
- Обычная иконка — 24, stroke width 2, rounded outline style.
- Анимации используют Reanimated и учитывают Reduce Motion.
- Нижнее меню всегда `Главная · События · Чаты · Клубы · Профиль`.
- Экран проверяется на iOS, Android, ширине 320–430 и при 200% text scaling.

## Аккаунты и профили

- Не путать `user_account` и `actor_profile`.
- Один аккаунт может иметь player, trainer и organization actor-профили.
- Профиль игрока существует всегда.
- Все создаваемые сущности хранят `created_by_user_id`, `created_by_actor_id`, `created_by_actor_type`.
- Мастер создания показывает `Создать как`.
- Переключение actor пересчитывает разрешения и адаптирует интерфейс, но не меняет аккаунт.
- Actor type сам по себе не даёт права управлять чужой сущностью.

## Игроки и приглашения

- Единственное пользовательское понятие для спортивного человека — `Игрок`.
- Каждый actor имеет собственный односторонний список `Мои игроки`.
- Не создавать сущности `Друзья`, `Ученики`, `Партнёры`, `Подписчики` или социальные группы.
- Добавление в `Мои игроки` не требует подтверждения и не создаёт участие.
- Для выбора людей во всех сущностях использовать один player picker: `Мои игроки · Недавние · Поиск`.
- `Профиль → Мои игроки` является profile stack-экраном: глобальное нижнее меню сохраняется, активна вкладка `Профиль`.
- `player.picker` является отдельным full-screen modal из конкретного события: нижнее меню скрыто, обязательны entity/draft context и `returnTo`.
- Видимость применяется ко всей сущности: `Публичная` или `Непубличная`; не публиковать отдельно свободные места.
- Публичная сущность появляется в `События → Все`; непубличная скрыта из `Все`, но доступна в `Участвую` и `Управляю` по отношению пользователя.
- Способ набора `open / request / invitation_only` не смешивать с видимостью.
- Приглашение всегда относится к конкретному событию.
- Приглашённый пользователь не получает доступ к чату до подтверждения участия.
- Добавление игрока как подтверждённого после офлайн-договорённости требует разрешения и audit event.
- Player directory не является основанием для доступа к закрытым данным игрока.

## Навигация

- Пользовательские названия нижних вкладок: `Главная · События · Чаты · Клубы · Профиль`.
- Внутренний id второй вкладки остаётся `play`, маршрут — `/play`.
- Не создавать вторую нижнюю навигацию или отдельную оболочку внутри кабинетов.
- `События` всегда содержит вкладки `Все · Участвую · Управляю`.
- Игрок по умолчанию открывает `Участвую` или `Все`, тренер и организация при наличии рабочих задач — `Управляю`.
- Вкладка `Все` является публичным каталогом.
- Вкладка `Участвую` содержит личные участия, заявки, приглашения и лист ожидания.
- Вкладка `Управляю` содержит созданные и доступные по роли события, задачи и черновики.
- Структура раздела одинакова для всех actor-профилей; меняются порядок блоков, вкладка по умолчанию и permissions.
- Полный архив, настройки, финансы и статистика находятся в Профиле.
- Чат, уведомление и календарь ведут на канонический контекст.

## Чаты и разделы событий

- У каждой опубликованной игры, тренировки, турнира, сезона и тура есть один чат события.
- Базовые разделы канонического экрана: `Обзор`, `Игроки / Пары / Команды`, `Чат`.
- Расписание, раунды, сетка, таблица, результаты, статистика, программа, документы и оплаты подключаются по типу, формату, статусу и роли.
- Гость не видит сообщения чата.
- Подтверждённый участник, организатор и уполномоченный сотрудник получают доступ согласно политике.
- Не создавать отдельный групповой чат вне конкретного события.
- После завершения чат остаётся доступным, становится read-only или архивируется по политике; он не превращается в социальную группу.
