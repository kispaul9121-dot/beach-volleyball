# Аудит полноты архитектуры

Дата аудита: 17 июля 2026

## Итог до версии 2

Текущая документация правильно зафиксировала пять нижних вкладок, канонические экраны сущностей, разделение публичного каталога «Играть» и личного раздела «Профиль → Мои игры», а также базовый формат «ротация пяти игроков».

Однако архитектура была недостаточна для реализации приложения целиком.

## Критические пробелы

| Область | Состояние до аудита | Что добавляется в v2 |
|---|---|---|
| Вход и регистрация | отсутствовали | Google, email, подтверждение, восстановление, онбординг |
| Несколько профилей | роли были только списком прав | actor-профили игрока, тренера и организации, активный профиль |
| Создание от имени профиля | отсутствовало | обязательный `created_by_actor_id` и выбор «Создать как» |
| Главная и Играть для создателя | не было быстрого доступа | закреплённые блоки «Управляю» и «Мои ближайшие» |
| Мои турниры и сезоны | одна строка | участие, организация, черновики, действия, кабинеты |
| Мои тренировки | одна строка | игрок, тренер, группы, посещаемость, оплаты |
| Мои туры | одна строка | неформальные, тренерские и организационные туры |
| Статистика | отсутствовала | варианты игрока, тренера и организации |
| Настройки | отсутствовали | аккаунт, профили, безопасность, уведомления, приватность, выплаты |
| Форматы соревнований | перечислялись без алгоритма | конфигурации, таблицы, тай-брейки, раунды, экраны управления |
| Швейцарская система | отсутствовала | правила жеребьёвки, повторных встреч, bye и тай-брейков |
| Полное распределение | только упоминание 1–32 | ограничения, количество раундов, матчи и места |
| Внутренние экраны | описаны частично | создание, управление, матчи, игровые дни, оплаты, чат |
| Жизненные циклы | отсутствовали | статусы и разрешённые переходы сущностей |

## Критерий «экран продуман»

Экран считается архитектурно готовым, когда определены:

1. `screen_id` и маршрут;
2. доступные actor-профили;
3. назначение и источник данных;
4. основные блоки;
5. все кликабельные действия;
6. состояния loading, empty, error, offline и permission denied;
7. поведение для участника и организатора;
8. возврат назад и deep link;
9. связь с уведомлениями, чатом и календарём;
10. отдельная спецификация или явно описанный вариант общего экрана.

## Уровень готовности после обновления v2

После этого обновления архитектура будет готова для поэтапной реализации MVP, но не будет означать, что приложение уже работает. Для фактической работы ещё нужны:

- реализация интерфейса;
- база данных и политики доступа;
- алгоритмы жеребьёвки и таблиц;
- платёжная интеграция;
- тесты сценариев и расчётов;
- проверка UX на прототипах.

Документация v2 задаёт контракт, по которому Codex может реализовывать и проверять эти части.
