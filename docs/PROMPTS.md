# Промпты для Codex

## Реализация экрана

```text
Прочитай AGENTS.md и все обязательные документы, перечисленные в нём.
Реализуй screen_id: <screen_id>.

Используй существующий маршрут из ROUTES.yaml.
Все кликабельные элементы свяжи с action_id из ACTIONS.yaml.
Не создавай копии канонических экранов.
Реализуй варианты actor-профилей и отношения к сущности, указанные в SCREENS.yaml.
Добавь loading, empty, error, offline и permission denied состояния.
Проверь back_fallback и прямой deep link.
```

## Проверка навигации

```text
Проверь SCREENS.yaml, ACTIONS.yaml и ROUTES.yaml.

Найди:
- source или destination без существующего screen_id;
- маршрут без экрана;
- экран без маршрута, если это не system overlay;
- stack-экран без back_fallback;
- кнопку без action_id;
- дублирующие канонические страницы;
- manage route без permission;
- create route без actorId / returnTo поддержки;
- spec path, которого нет в репозитории.

Сначала составь отчёт, затем исправь документацию.
```

## Проверка actor-профилей

```text
Прочитай ACTORS.yaml, CAPABILITIES.yaml и ROLES.yaml.
Для выбранного сценария проверь player, trainer и organization.

Проверь:
- активный actor видим в шапке;
- разрешения пересчитываются после переключения;
- created_by_user_id и created_by_actor_id сохраняются;
- частное участие человека не смешивается с владением организации;
- финансовые ограничения блокируют только нужную функцию;
- у пользователя нет доступа к чужому кабинету управления.
```

## Реализация соревнования

```text
Прочитай COMPETITION_FORMATS.yaml, GAME_FORMATS.yaml,
ENTITY_LIFECYCLES.yaml, tournament-manage.md и competition-match.md.

Реализуй формат <format_id> как отдельную стратегию движка.
Не смешивай progression турнира и правила отдельного матча.
Храни результаты как события.
Опубликованные раунды версионируй.
Любой override сохраняй с причиной и actor-контекстом.
Добавь property-based или генеративные тесты инвариантов формата.
```

## Тест швейцарской системы

```text
Для Swiss создай тесты:
- нет повторной пары, пока существует валидная альтернатива;
- bye получает участник без предыдущего bye по заданному правилу;
- один участник не появляется дважды в раунде;
- следующий раунд не создаётся при незавершённых результатах;
- таблица использует заданный порядок тай-брейков;
- ручное изменение пары повторно валидируется.
```

## Тест полного распределения

```text
Для N = 4, 8, 16, 32 проверь:
- число раундов = log2(N);
- каждый участник играет один матч в каждом раунде;
- всего N/2 * log2(N) матчей;
- каждый получает одно уникальное место 1..N;
- победитель и проигравший переходят в правильные placement-ветки.
```

## Проверка UI участника и организатора

```text
Для сущности <entity_type> пройди три состояния:
1. посторонний;
2. участник;
3. организатор / сотрудник.

Проверь главное действие, видимые данные, оплату, чат, быстрый доступ,
переход в manage и отсутствие недоступных организаторских кнопок.
```

## Обновление архитектуры

Перед спорным изменением добавь решение в `DECISIONS.md`. Затем синхронно обнови:

- `SCREENS.yaml`;
- `ACTIONS.yaml`;
- `ROUTES.yaml`;
- actor / capability / format файлы, если затронуты;
- спецификацию экрана;
- `APP_MAP.md` и Mermaid-карту.
