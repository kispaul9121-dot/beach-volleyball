# VolleyPlay — runbook последовательной реализации

Этот каталог содержит **80 непересекающихся промтов** для поэтапной реализации VolleyPlay на основе архитектуры репозитория.

## Как использовать

1. Выполнять промты строго по номеру, если в самом промте не написано обратное.
2. Один промт — один отдельный рабочий сеанс coding-agent и один логический commit.
3. Перед началом каждого промта агент обязан прочитать `AGENTS.md`, этот runbook и перечисленные в промте источники.
4. Перед изменением файлов агент проверяет фактическое состояние текущей ветки. Уже корректно реализованное не переписывается.
5. После каждого блока реализации запускается следующий контрольный промт. Контрольный промт не добавляет новые продуктовые возможности.
6. Не выполнять merge, rebase, force-push, удаление чужой работы или массовое форматирование без отдельного указания пользователя.

## Источники истины

Приоритет источников:

1. Последние явно утверждённые продуктовые решения в `docs/DECISIONS.md`.
2. Машиночитаемые контракты `docs/*.yaml`.
3. Спецификации `docs/screens/**/*.md`.
4. `docs/ARCHITECTURE.md`, `docs/DATA_MODEL.md`, `docs/DESIGN_SYSTEM.md`.
5. Реальный работающий код и тесты.

Если источники противоречат друг другу, агент не угадывает. Он останавливает затронутую часть, фиксирует конфликт в отчёте и меняет только то, что прямо разрешено текущим промтом. Исключение — промт `001`, который специально синхронизирует утверждённый дизайн.

## Статус-зависимая работа

Перед каждой задачей определить состояние области:

- `implemented` — реализовано и подтверждено тестами;
- `partial` — работает часть сценария;
- `placeholder` — есть маршрут или оболочка без полной функции;
- `definition_pending` — продуктовый контракт не завершён;
- `not_started` — реализации нет;
- `blocked` — есть явный конфликт или внешняя зависимость.

Правила:

- `implemented`: не переписывать; добавить только отсутствующую проверку или точечное исправление.
- `partial`: сохранить работающее поведение и дополнить только область текущего промта.
- `placeholder`: оставить совместимый API и заменить только разрешённую часть.
- `definition_pending`: не придумывать бизнес-правила; создать интерфейс, типы, feature flag или честный placeholder для будущей работы.
- `blocked`: не маскировать проблему mock-логикой; описать блокировку и безопасную основу.

Каждый промт должен обновлять `docs/IMPLEMENTATION_STATUS.yaml`, когда фактический статус области изменился. Если файла ещё нет, его создаёт промт `002`.

## Неповторяемость

Общие правила не копируются в каждом промте. Каждый промт имеет собственную область ответственности и не должен:

- переделывать результат предыдущего промта без доказанной ошибки;
- расширять продуктовый scope;
- менять несвязанные маршруты, роли или форматы;
- создавать второй канонический экран;
- создавать локальные копии общих UI-компонентов;
- «заодно» рефакторить соседние модули.

При необходимой межмодульной правке агент делает минимальное изменение и перечисляет его в отчёте.

## Общий формат завершения каждого промта

В конце агент обязан вывести:

1. фактический статус до и после;
2. изменённые файлы;
3. реализованный пользовательский сценарий;
4. добавленные тесты;
5. выполненные команды и их результат;
6. оставшиеся честные ограничения и TODO;
7. SHA логического commit, если commit разрешён средой.

## Проверки реализации

Минимально применимые проверки:

- typecheck;
- lint;
- unit tests;
- component/integration tests;
- route/action validators из `scripts/`;
- проверка прав отрицательными тестами;
- loading, empty, error, offline и permission states;
- iOS и Android;
- ширины 320, 360, 390 и 430;
- 200% text scaling;
- VoiceOver/TalkBack semantics;
- Reduce Motion;
- отсутствие hard-coded цветов в экранах;
- отсутствие второго чата, второго канонического detail screen или второй нижней навигации.

## Утверждённое визуальное направление

Промт `001` должен синхронизировать репозиторий со следующим решением:

- классический спортивный **light-first** интерфейс;
- белые и очень светлые поверхности;
- основной синий акцент;
- тёмно-синий основной текст;
- нейтральные серо-синие границы;
- зелёный — успех;
- янтарный — ожидание и предупреждение;
- красный — ошибка и опасное действие;
- голубой — информация;
- dark и system режимы архитектурно поддерживаются, но светлая тема является основной;
- конкретные значения находятся только в `DESIGN_TOKENS.yaml`; экраны используют semantic tokens.

## Граница MVP

Пакет не возвращает удалённые решения:

- сезоны и повторяющиеся игровые дни;
- круговую и швейцарскую системы турниров;
- `Группы + плей-офф`;
- отдельный турнир `Король пляжа`;
- двойное выбывание без нового продуктового решения.

Для турниров остаются `single_elimination` и `full_placement`. Для полного распределения 32 команды проводят 80 матчей. `Король пляжа`, `Тунисский формат` и `Смешанные пары` разрешаются как пользовательские названия разовой `Тунисской лестницы`.
