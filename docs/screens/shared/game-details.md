# Страница игры — черновой визуальный контракт

- Screen ID: `game.details`
- Route: `/games/:gameId`
- Variants: `guest`, `participant`, `organizer`
- Visual product status: `definition_pending`
- Join contract: `docs/JOIN_FLOW.yaml`
- Invitation contract: `docs/INVITATION_DISCOVERY.yaml`
- Game MVP contract: `docs/GAME_MVP.yaml`

## Статус решения

Визуальная композиция страницы игры пока не утверждена продуктово. Текущий документ фиксирует технический контракт и уже утверждённые механики вступления, оплаты, состава, счёта и чата.

До отдельного визуального решения нельзя считать финальными:

- структуру верхней части экрана;
- точный порядок внутренних разделов для участника;
- финальную мобильную компоновку и дизайн.

Управленческий экран уже использует разделы `Обзор · Участники · Счёт · Чат`. Участник получает доступ к тем же данным по своим правам, но без управленческих действий.

## Зафиксированный минимум

Экран должен уметь получить и показать:

- идентификатор и статус игры;
- название и формат;
- создателя и actor-тип;
- дату, время и площадку;
- публичность и способ вступления;
- количество игроков, занятые и свободные места;
- лимит выбранной площадки;
- цену либо статус `Бесплатно`;
- отношение текущего пользователя: посторонний, приглашённый, заявитель, участник или организатор;
- отмену, перенос или недоступность игры;
- безопасный возврат в исходный список с восстановлением фильтров и позиции.

## Утверждённая механика вступления

Главное действие для пользователя без отношения к игре называется:

```text
Вступить
```

Resolver использует две независимые настройки организатора:

```text
Как игроки вступают?
Сразу записываются · Отправляют заявку · Только по приглашению
```

```text
Стоимость
Бесплатно · Оплата онлайн · Оплата организатору вне приложения
```

Результат определяется сервером:

- свободное место + мгновенная бесплатная запись → `confirmed`;
- свободное место + онлайн-оплата → `payment_required`;
- заявка → `requested`, а оплата при необходимости запрашивается после одобрения;
- заполненная игра → `waitlisted`, если лист ожидания включён;
- `Только по приглашению` без активного приглашения → кнопка вступления скрыта и показывается объяснение;
- активное приглашение открывается через `/invitations/:invitationId`.

Нерешённое приглашение ещё не является участием и не показывается в Профиле. После создания статуса участия запускается одноразовый сигнал на иконке Профиля.

## Состав и оплата для подтверждённого участника

Подтверждённый участник видит список игроков с именем и фамилией. Для остальных показывается только статус:

```text
Оплачено
Не оплачено
Бесплатно
```

Кнопок возле чужих строк нет.

В собственной строке при необходимой онлайн-оплате показывается:

```text
Оплатить
```

Кнопка использует `entity.open_payment` и открывает checkout только для собственного payment record. После успешной оплаты она заменяется на статус `Оплачено`, такой же, как у остальных игроков. Во время обработки показывается `Оплата обрабатывается`, после ошибки — `Повторить оплату`.

Платить за другого участника из состава запрещено. Гости, приглашённые и пользователи с необработанной заявкой не видят платёжные статусы состава.

## Счёт и статистика

Подтверждённый участник видит опубликованный счёт и статистику в представлении, зависящем от формата:

- `2×2`, `4×4`, фиксированные команды — таблица сетов;
- ротация пяти — карточки раундов и кортов плюс личная таблица результатов.

Права ввода и исправления счёта определяются отдельно; обычный участник не получает их автоматически.

## Чат

Подтверждённый участник видит раздел `Чат`. Это тот же conversation, который находится в общей вкладке `Чаты`; отдельная копия истории не создаётся.

Приглашённый пользователь и заявитель не получают доступ к чату до подтверждения участия.

## Канонические входы

Один и тот же маршрут используется из:

- каталога `Игры`;
- личной активности;
- уведомлений;
- чата игры;
- календаря;
- публичного профиля игрока или организации, когда там показана игра.

Приглашение сначала открывает `invitation.details`, а после принятия может вести сюда уже в статусе участника.

Отдельные копии страницы игры для разных точек входа запрещены.

## Доступ и действия

- публичный preview может быть доступен гостю;
- непубличная игра открывается только при допустимом отношении или разрешении;
- изменение состава, параметров и результатов выполняется через `game.manage`;
- участие, заявка, приглашение и оплата используют `docs/JOIN_FLOW.yaml`;
- чат доступен подтверждённым участникам согласно `docs/ENTITY_SECTIONS.yaml`;
- нижняя навигация приложения не заменяется локальным меню.

## Состояния, которые нельзя потерять при проектировании

- loading;
- игра отменена;
- дата или площадка изменены;
- регистрация закрыта;
- мест нет;
- заявка ожидает решения;
- пользователь в листе ожидания;
- требуется оплата;
- оплата обрабатывается;
- оплата не прошла;
- подтверждённое участие;
- пользователь удалён организатором;
- потеряно право управления;
- offline read-only;
- ошибка загрузки или устаревший deep link.

## Следующее визуальное решение

Перед реализацией требуется отдельно показать страницу минимум в четырёх состояниях: открытая игра для постороннего, приглашённый игрок, подтверждённый участник и организатор. Утверждённая механика собственной оплаты в списке участников не должна переопределяться макетом.

## Открытие из каталога по правам

При нажатии на обычную карточку в `Игры` публичный экран открывается пользователю без действующего права управления. Для manager actor resolver сразу открывает manage route. При прямом deep link владелец видит действие `Управлять`, но публичный и управленческий режимы не смешиваются.
